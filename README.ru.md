# 🧭 AI Engineering Standard (AES)

> **Стандарт разработки программного обеспечения с AI‑агентами.**

**AI Engineering Standard (AES)** — это системный подход к совместной работе человека и AI при разработке ПО.

AES превращает AI из генератора кода в **предсказуемого инженерного партнёра**, работающего по контракту, а не по промптам.

---

<p align="center">
<strong>От промптов → к инженерии.</strong>
</p>

---

## 🌍 Почему появился AES

Разработка с AI без структуры приводит к:

- хаотичной архитектуре
- потере контекста между сессиями
- случайным регрессиям
- несогласованным решениям
- постоянным переписываниям

AES вводит **контракты, состояния и правила исполнения**, превращая AI‑разработку в управляемый инженерный процесс.

---

## 🧠 Главный принцип

> AI исполняет решения, а не придумывает их.

| Роль | Ответственность |
|------|-----------------|
| Человек | продукт, архитектура, приоритеты |
| AI‑агент | анализ, реализация, проверка |

---

## 🧩 Основные контракты AES

AES состоит из двух обязательных файлов.

### 1️⃣ PROJECT_CONSTITUTION.md — продуктовый контракт

Определяет:
- scope и non‑goals
- архитектуру
- стек технологий
- доменную модель
- безопасность
- деплой
- критерии готовности

➡️ **Что мы строим?**

---

### 2️⃣ CLAUDE.md — контракт исполнения AI

`CLAUDE.md` — ключевой файл AES.

Он автоматически читается AI‑агентом и определяет:

- роль AI как senior‑разработчика
- жизненный цикл разработки
- допустимые и запрещённые практики
- систему команд
- условия остановки агента
- инженерные требования

➡️ **Как AI обязан это строить?**

---

## 🤖 Почему CLAUDE.md на английском

`CLAUDE.md` — это **machine‑readable execution specification**, а не пользовательская документация.

AI‑агенты обучены преимущественно на английском языке, поэтому:

- обеспечивается стабильное поведение
- команды выполняются предсказуемо
- сохраняется совместимость между AI IDE

Файл является **canonical execution contract** и не переводится.

## ⚙️ Стандартные команды агента

| Команда | Поведение |
|----------|-----------|
| `/constitution` | Создание конституции проекта |
| `/explore` | Анализ репозитория |
| `/plan` | План реализации |
| `/status` | Статус проекта |
| `/fix` | Поиск и исправление ошибок |
| `/review` | Code review |
| `/deploy` | Подготовка к деплою |

AI **не начинает реализацию без подтверждённого плана**.

## 🔄 Жизненный цикл разработки (AES)

### Phase 1 — Alignment
- анализ проекта
- формирование конституции
- проверка контекста

### Phase 2 — Planning
- пошаговый план
- выявление рисков
- подтверждение человеком

### Phase 3 — Execution
- итеративная реализация
- соблюдение архитектуры
- постоянная проверка

### Phase 4 — Delivery
- проверка стабильности
- очистка debug‑кода
- готовность к продакшену

## 🧱 Инженерные требования

### Качество кода
- строгая типизация
- один файл — одна ответственность
- ≤150 строк на модуль
- осмысленные имена

### Архитектура
- разделение слоёв
- изоляция зависимостей
- конфигурация через env

### Надёжность
- обработка внешних вызовов
- graceful degradation
- логирование
- восстановление после ошибок

### UX‑база
- loading state
- empty state
- error state
- подтверждение опасных действий

---

## ⛔ Нарушения стандарта AES

- подавление типизации (`any`, `ts-ignore`)
- хардкод конфигурации
- debug‑логи в проде
- мёртвый код
- пустые catch
- дублирование логики
- TODO без описания

## ✅ Уровни соответствия AES

| Уровень | Описание |
|----------|----------|
| AES‑L1 | Определена конституция |
| AES‑L2 | Есть план реализации |
| AES‑L3 | Соблюдена архитектура |
| AES‑L4 | Production Ready |

Проект может указывать:

```
AES: L3 Compliant
```
